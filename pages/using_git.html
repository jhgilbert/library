Using Git

About version control

If you've done much programming, you know that false starts are inevitable. Backtracking once you've developed your code in the wrong direction can be painful. That's where version control comes in.

From here on out, everything program you write or website you build should be in version control. Git is the program you'll use locally (on your computer). GitHub is a separate but useful service that stores Git repositories online for you -- kind of like Flickr, but for repositories.

It's easy to get Git and GitHub confused or treat them as the same thing, but they aren't. You don't have to use GitHub -- other services, like Bitbucket, are available. But GitHub is a common industry standard, so we'll go with that.

A few advantages of version control:

  • Git creates save points (known as commits) that you can roll back to.
  • Git allows you to branch off from your code, so you can try a few different approaches simultaneously and see which works better.
  • If you use GitHub to store your code, it's backed up safely and you won't lose it.

IMPORTANT: Unless you have a paid subscription with GitHub, all of your repositories are viewable to the public.

It's fine to build a fun project for yourself involving kittens and magic sorcerers (the programming community loves creativity!) but even the whimsical stuff should be kept relatively professional. Even if you're just writing test copy, don't make offensive or personal jokes that you wouldn't want a potential boss to see.

The basic Git cycle

Your most common workflow will go like this:

  1. Run git pull to make sure you're in sync with GitHub (in other words, that you have the most current code, as another developer could have changed it elsewhere)
  2. Create a new file, or modify an existing one
  3. Use git add [filename] in Terminal to add the changed/new file to the set of files you intend to commit (this is also known as staging the file)
  4. Use git commit -m "commit message" to create the actual save point
  5. Optionally, use git push to push your changes up to GitHub for online storage, but most programmers just do this periodically, rather than every time

That's the basic loop you'll go through over and over, but you might also occasionally use other commands to recover from a mistake. We'll practice the basic commands first, then look at a few special cases.

When you decide to build a complex project, it should have its own Git repo. But for now, your entire web projects folder is one big repo. (So are each of your other folders, such as your database folder, etc.)

In other words, if you visited GitHub right now (try it!), you'd see that you have four repos -- one for each of your folders in /projects.

Because any websites you make at this stage are likely simply for learning purposes, keeping several small projects in one Git repo is fine. But if you decide to build a real-world website that will actually see the light of day, put it in its own dedicated repository. Instructions for doing that appear at the bottom of this page.

Some considerations

What dangerous commands should I avoid as a beginner?

reset and force should be used very carefully. If you're trying to solve an issue and the solution you find online involves those commands (especially force), ask for help unless you're sure you know what you're doing.

When shouldn't I add a file to my repository?

Consider whether your repository is available to the public. If it is, avoid committing any configuration files that include passwords or other sensitive information.

Depending on your operating system and the nature of your project, other files may pop up that you should avoid committing. On a Mac, for instance, it's often considered best practice to avoid committing .DS_Store files (which are automatically created by the operating system).

It will take a while to get the hang of how all of this works. In the short term, focus on committing only the files that you have created or modified yourself. If you type 'git status' and see an untracked file you don't recognize, it may have been automatically created by your computer and may not belong in a repo.

If you would like to ignore files of a certain type, you can create a file called .gitignore in your directory, and add the files you'd like to avoid. You can then add and commit the .gitignore file.

A sample .gitignore file that could be put in any given directory on a Mac machine to avoid accidentally committing .DS_Store files into the repo:

projects/.gitignore
## Files to ignore, using * as a wildcard match
*.DS_Store

If you'd like to practice, feel free to create the file above in Notepad++ and commit it to your repo.

How do I access an older version of my code?

Several methods will work. If you go on GitHub, you'll see that each commit has a unique identifier (something long and complicated, like 5da5981a813eb257e027cf38662502b5930cd0f3). git checkout [commit identifier] will take you back in time -- your project folder will look just as it did at the time, and your code will run as it did at the time. The exception is that any uncommitted local files you had hanging around in your folder will still be present.

How often should I commit?

You should commit relatively frequently. At some point, you might break your code and not notice for quite a while. With Git, it's easy to roll back to previous commits until you find one that isn't broken. If your commits are small, you'll then find it quite easy to review the bad commit to figure out where you went wrong.

If the bad commit has 5,000 changed lines in it? Not so much. Have fun finding the problem.

Commit anytime you get to a logical stopping place -- maybe you've just defined a new method or finished writing a loop. Don't get so into your code that you forget to commit for quite a long time -- you may regret it!

If I run git status, I see a line that says "git branch master". What does that mean?

The master branch is the default branch of a repo. Even though we just did it in the above exercise, it's actually not a great idea to work on the master branch. Most new programmers work on the master branch for the sake of simplicity (and because they're not working with a team), but at a real company, the master branch often contains the code for the actual deployed application. You can see how it might be a bad idea to start committing to that code when you aren't even sure you're headed in the right direction with an idea.

For safety reasons, and to avoid crashing into one another in terms of working on the same code at the same time, developers will typically branch off from master, develop a feature, and merge the code back into master when the feature is complete, tested, and ready to go.

If this idea confuses you, think of it as pulling off the racetrack to fix up your car, rather than trying to fix it while still racing on the track. It's safer to remove yourself from the race, improve your car (or section of the website), verify that everything is running smoothly, and then merge back in.

For now, it's up to you whether to work on master. Instructions for getting fancy and creating your own branch are at the bottom of this page.

I made a mistake in my last commit and want to fix it.

You can amend a commit by making the desired changes, adding the modified files to staging, and then running git commit --amend -m "New commit message". Git has a ton of tricks like that -- almost too many!

How to Branch

As mentioned earlier, it's best to make a new branch anytime you're doing new work on an existing project. This involves just a few additional steps.

  1. Go to the directory of the project you'd like to work on
  2. Run git checkout -b new_branch_name to create and check out your new branch -- the name can reflect whatever you intend to do, such as css_tweaks or database_upgrades.
  3. Run git status to verify that you are on the desired branch.
  4. Work on your project, adding and committing files as you go. If you want to push your new branch to GitHub at any point, git push origin new_branch_name will take care of that.
  5. When you've reached a good stopping place and everything is working well, you can merge back into master (or whichever branch you're merging back into) by running git checkout master to check out the master branch, and then git merge new_branch_name to pull in changes.
  6. Running git branch will show you a list of branches, with a star by the name of the one you are currently on.
  7. If you've finished with your new branch and would like to delete it, run git branch -D new_branch_name to clean up after yourself. (If you pushed to GitHub, you will need to delete the branch on GitHub as well.) That's it!
